# Spin up a new SaaS vertical in 30 minutes

This walkthrough takes you from zero to a four-app SaaS skeleton (domain + BFF + admin SPA + marketing site) for any new vertical — jobs board, real-estate site, classifieds, etc. — in about half an hour.

You'll get:

- A Spring Boot BFF with users + auth + image upload + your domain CRUD
- A React SPA with admin pages auto-generated from the domain spec, plus dashboard, sign-in/up flows, and image gallery
- A static Next.js marketing site with a pricing page
- One YAML file per app — `fixedcode build` produces the rest

## Prerequisites

- Node 20+
- JDK 21 (for the BFF)
- `fixedcode` CLI installed: `npm install -g fixedcode`

## 1. Scaffold the project (1 min)

```bash
fixedcode init saas-vertical jobs
cd jobs
ls
```

You'll see:

```
jobs/
├── .fixedcode.yaml          # registers the bundles your specs use
├── README.md                # generated quickstart
└── specs/
    ├── jobs-domain.yaml     # entities & operations (drives BE CRUD + FE admin pages)
    ├── jobs-bff.yaml        # Kotlin/Spring BFF with users + image-upload recipes
    ├── jobs-app.yaml        # Vite/React SPA with admin-screen + dashboard + users + image
    └── jobs-marketing.yaml  # Next.js static marketing site with pricing recipe
```

The four specs reference each other where it matters. The SPA's `admin-screen` recipe reads the domain spec; the BFF's users recipe is independent; both halves of `users-management` share JWT tokens.

## 2. Edit the domain spec (5 min)

Open `specs/jobs-domain.yaml`. The starter has two example aggregates (`Job`, `Application`). Edit them to model your vertical:

```yaml
spec:
  aggregates:
    Job:
      attributes:
        - { name: id, type: uuid }
        - { name: title, type: string }
        - { name: company, type: string }
        - { name: salary, type: int }
        - { name: location, type: string }
        - { name: postedAt, type: instant }
      commands: [Create, Update, Delete]
      queries: [Get, Search, FindByCompany]
    Application:
      attributes:
        - { name: id, type: uuid }
        - { name: jobId, type: uuid }
        - { name: candidate, type: string }
        - { name: status, type: string }
      commands: [Create, UpdateStatus]
      queries: [Get, FindByJob]
```

Field types map to BE Kotlin types AND to FE form inputs (`string` → `<TextField>`, `int` → `<NumberField>`, `instant` → `<DateTimeField>`, etc.). Add or remove aggregates freely.

## 3. Wire the bundles (1 min)

If you're inside the fixedcode monorepo, edit `.fixedcode.yaml` and replace each `@fixedcode/bundle-*` reference with the local path:

```yaml
bundles:
  spring-domain:       './../bundles/spring-domain'
  kotlin-spring-bff:   './../bundles/kotlin-spring-bff'
  vite-react-app:      './../bundles/vite-react-app'
  next-marketing-site: './../bundles/next-marketing-site'
```

(When the bundles ship to npm, this swap goes away — the `@fixedcode/...` references will resolve directly.)

## 4. Generate everything (10 sec)

```bash
fixedcode build specs/ -o build
```

You now have four built apps under `build/`:

```
build/
├── jobs-domain/      # generated by spring-domain — Kotlin entities, repos, controllers, OpenAPI
├── jobs-bff/         # Spring Boot BFF — auth, users, image upload, your domain endpoints
├── jobs-app/         # Vite/React SPA — admin pages per aggregate, dashboard, sign-in/up
└── jobs-marketing/   # Next.js static — landing, pricing, about
```

## 5. Run the BFF (3 min)

```bash
cd build/jobs-bff
./gradlew bootRun
```

Default port: 8090. The Flyway migration creates the `users` table. The first time you sign up via the FE, the user gets `ROLE_USER`. Promote yourself to `ROLE_ADMIN` directly in the DB to access admin endpoints:

```sql
UPDATE users SET role = 'ADMIN' WHERE email = 'you@example.com';
```

## 6. Run the SPA (2 min)

```bash
cd ../jobs-app
npm install
npm run dev
```

Open <http://localhost:5173>. Sign up. The bundled routes:

- `/` — landing
- `/sign-in`, `/sign-up` — auth
- `/admin` — dashboard (stat cards)
- `/admin/jobs` — list + create + edit (auto-generated from your spec)
- `/admin/applications` — same, for the second aggregate

Edit the extension-point files (`src/admin/.../*Form.tsx`, `src/auth/AuthProvider.tsx`, etc.) to customise. They're flagged `overwrite: false` so regeneration won't clobber your work.

## 7. Run the marketing site (2 min)

```bash
cd ../jobs-marketing
npm install
npm run dev
```

Open <http://localhost:3000>. Edit `specs/jobs-marketing.yaml` and re-run `fixedcode build` to update brand text, hero copy, and pricing tiers.

## 8. Iterate

The whole point: when the spec changes, regenerate. Generated files (`overwrite: true`) get rewritten; extension points (`overwrite: false`) are preserved.

```bash
# add a new aggregate or field to specs/jobs-domain.yaml
fixedcode build specs/ -o build
```

Your custom business logic and form tweaks survive. The new entity's controller, repository, list page, and admin sidebar link all appear automatically.

## What's NOT included

- A charting library on the dashboard — `dashboard` ships the layout + stat cards. Wire Recharts / Tremor / Chart.js in the extension points when you need real visualisations.
- Real payment processing on the pricing page — tiers are display-only. Wire Stripe yourself; it's too domain-specific to bundle.
- A CMS for marketing copy — copy lives in the YAML spec. For longer-form content (blog), add MDX to the next-marketing-site extension points.
- Production deploy — `fixedcode deploy` copies into a target project; for cloud deploy, the BFF has a generated `Dockerfile` (`features.docker: true`), the SPA can target Cloudflare Pages / Vercel / Netlify, and the marketing site is static-export ready.

## Cheat sheet

| Need | Where |
|---|---|
| Add a new entity | `specs/<name>-domain.yaml` → regenerate |
| Add a new admin page | (automatic — generated from domain spec) |
| Customise a form | `src/admin/<entity>/<Entity>Form.tsx` (extension point) |
| Change auth provider | `src/auth/AuthProvider.tsx` + `JwtService.kt` (both extension points) |
| Change pricing tiers | `specs/<name>-marketing.yaml` → regenerate |
| Add a stat to the dashboard | `specs/<name>-app.yaml` → `dashboard.stats: [...]` → regenerate |
| Swap image storage to S3 | `service/LocalImageService.kt` (extension point) → implement against AWS SDK |
